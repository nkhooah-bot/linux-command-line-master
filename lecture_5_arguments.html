Section 1Command Arguments: Context & Constraints

To guide shell applications safely, we pass metadata strings known as arguments alongside the target binary name. An argument provides necessary context to an executing command, specifying exactly what object or address the utility should work on.

Recall our previous diagnostic routine: ping google.com. Here, ping represents the program application, while google.com serves as the target argument.

Arguments are categorized into two structural profiles depending on the requirements of the application:

  • Mandatory Arguments: Information required by the application to execute. For example, the ping command requires a target address argument, as it cannot run without knowing which remote server to communicate with.
  • Optional Arguments: Information that can be omitted, allowing the program to fallback to default behaviors or run without an explicit target.

Section 2The Process Status (ps) Utility Engine

Many standard commands do not require any positional arguments to operate. A prime example is the ps (Process Status) command.

In a Linux system, a process is an active, running instance of a program being executed by the operating system kernel. Typing ps on its own without any arguments outputs a list of the active processes tied to your current terminal session:

user@ubuntu-server:~$ ps PID TTY TIME CMD 1432 pts/0 00:00:00 bash 1547 pts/0 00:00:00 ps

To interpret these system output columns properly, look at the tracking headers defined below:

  1. PID (Process Identifier): A unique tracking index number assigned by the operating system kernel to isolate each running program instance.
  2. TTY (Teletypewriter): Indicates the specific terminal window session associated with the execution space of that program.
  3. TIME: The total processor runtime clock duration utilized by the process since it was spawned.
  4. CMD (Command): The exact initiating binary command word typed to execute the application space.

Section 3Introduction to Options, Flags, and Switches

Even when an application requires no arguments, its core behavior can be highly altered using options (frequently called flags or switches). Options are specific characters or long-form phrases appended to a command to modify how the application behaves.

By default, the standard ps command filters out system background tasks, only showing applications running directly inside your active window interface. To override this and see every active system task, append the -e option, where the lowercase e stands for everything:

user@ubuntu-server:~$ ps -e PID TTY TIME CMD 1 ? 00:00:02 systemd 2 ? 00:00:00 kthreadd 1432 pts/0 00:00:00 bash 1561 pts/0 00:00:00 ps

Standard short-form options are prefixed with a single hyphen symbol (-). This marker tells the shell interpreter that the trailing letter is an action-modifying flag rather than a literal context argument string.

Options are not universal; a flag that modifies one program can mean something entirely different to another command, or might not be recognized at all. They are also strictly case-sensitive. Running ps -E will fail with an error because the application does not recognize an uppercase -E modifier:

user@ubuntu-server:~$ ps -E error: unknown option Usage: ps [options]

Section 4Combining Modifiers & Multi-Flag Cluster Syntax

To customize program outputs further, you can combine multiple options within a single command execution statement. For instance, you can combine the -e (everything) flag with the -f option, which stands for full-format documentation. The full-format option expands your process lists to include detailed operational columns like parent ownership IDs and precise execution timestamps.

Linux supports two different layout syntaxes for specifying multiple options:

Explicit Separate Syntax

Listing each flag distinctly, separated by spaces, with each letter maintaining its own individual hyphen prefix string.

user@ubuntu-server:~$ ps -e -f

Clustered Combined Syntax

Grouping multiple flag characters together behind a single hyphen. This produces a cleaner statement that is easier to read.

user@ubuntu-server:~$ ps -ef

Both structural layouts produce the exact same full system trace details. Choose the clustered combined layout to keep your manual typing concise.

Section 5Advanced Syntax: Passing Arguments to Options

Certain complex options require additional instructions to function. These are known as option arguments or parameters.

Consider the -u (user) option inside the process status engine. This flag targets and filters tasks based on account ownership. To use it, you must provide a target username immediately after the option flag:

user@ubuntu-server:~$ ps -u user PID TTY TIME CMD 1432 pts/0 00:00:00 bash

Many commands also support descriptive long-form options. Long-form flags use a full descriptive word instead of a single letter and are prefixed with two hyphens (--). For example, replacing the short -u option with its long-form --user equivalent works perfectly:

user@ubuntu-server:~$ ps --user user PID TTY TIME CMD 1432 pts/0 00:00:00 bash

Section 6Hands-on Laboratory: Storage Diagnostics Challenge

Instead of just memorizing syntax layouts, practice applying these patterns directly inside your active terminal sandbox using this disk monitoring workspace exercise:

Your Mission: Storage Volume Analysis

  1. Type the command df (Disk File System) on its own and press enter. Analyze the raw numeric block allocations printed across your device output.
  2. Modify your instruction by appending the human-readable option flag: df -h.
  3. Compare the two outputs. Notice how the -h option recalculates cryptic raw blocks into clean, human-readable storage sizes (like Megabytes M and Gigabytes G).

Get comfortable tracking how option flags alter your command outputs. Once your storage metrics challenge runs successfully, move forward to our next chapter, where we will learn how to read built-in system documentation using the terminal manual pages. See you there!
